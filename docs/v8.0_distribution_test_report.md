## 2. 데이터 모델 검증

| 모델 | 테이블명 | 주요 컬럼 |
|------|----------|-----------|
| `Vehicle` | `vehicles` | plate_no, driver_name, max_weight, is_active |
| `SalesOrder` | `sales_orders` | vendor_id(FK→vendors), status(ENUM 6종), total_amount |
| `SalesOrderItem` | `sales_order_items` | order_id, item_id(FK→items, nullable), item_name, quantity, unit_price |
| `Delivery` | `deliveries` | sales_order_id(FK, nullable), vehicle_id(FK, nullable), status(ENUM 5종), delivery_fee |
| `DeliveryReturn` | `delivery_returns` | delivery_id(FK), item_id(FK, nullable), reason(ENUM 5종), is_restocked |

### 자동화 동작
- **배송 완료 처리 시**: 연결 수주 `status` → 완료 자동 변경
- **반품 재고 복원 시**: `Item.current_stock` 증가 + `InventoryLog(반품)` 자동 생성
- **수주 총액**: 등록 시 품목 라인 합계 자동 계산

## 5. 알려진 제한사항

| 항목 | 내용 | 권장 해결책 |
|------|------|------------|
| 경로 최적화 알고리즘 | 지역 = 배송지 앞 3자 추출 (단순 문자열 기반) | 주소 파싱 라이브러리 적용 권장 |
| 배송 실시간 추적 | GPS 연동 없음, 수동 상태 변경 | 외부 배송 API 연동 고려 (v9.x) |
| 차량 과부하 체크 | max_weight 입력 가능하나 배송 시 초과 검증 없음 | 배송 등록 시 적재량 합계 검증 추가 권장 |
| 수주 수정 | 품목 라인 수정 엔드포인트 미구현 | 삭제 후 재등록으로 대체 가능 |
