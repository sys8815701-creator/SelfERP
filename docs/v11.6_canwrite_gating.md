# v11.6 구현 보고서 — 쓰기 · 삭제 권한 표시 누락 수정

**작성일**: 2026-07-21  
**범위**: employee 역할에서 admin · accountant 전용 조작이 그대로 노출되던 7개 화면에 `canWrite` · `canDelete` 게이팅 적용

---

### 1. 변경 사항

| 페이지 | 이전 | 이후 |
|---|---|---|
| `accounting/ar-ap` | 등록 버튼 · 상태 변경 셀렉트 · 수정 · 삭제 전 역할에 노출 | 등록 · 수정은 `canWrite`, 삭제는 `canDelete`, 상태는 읽기 전용 조회일 때 텍스트로 대체 |
| `accounting/tax-invoice` | 동일 | 동일 패턴 적용 |
| `accounting/estimates` | 목록 · 상세 드로어 양쪽에서 노출 | 목록 · 드로어 모두 동일 패턴 적용 |
| `accounting/budget` | 예산 항목 등록 · 수정 · 삭제 버튼 전 역할 노출 | 등록은 `canWrite`, 삭제는 `canDelete`, 항목 목록의 조작 열 자체를 조건부 렌더 |
| `hr/departments` | 부서 · 직급 추가 · 수정 · 삭제 버튼 전 역할 노출 | 두 탭 모두 동일 패턴 적용 |
| `production/inventory` | 입출고 등록 버튼 전 역할 노출(수정 · 삭제 기능 자체는 없음) | 등록 버튼만 `canWrite`로 제한 |
| `card` | 주 거래 은행 등록 · 수정, 거래 내역 수정 · 삭제 버튼 전 역할 노출 | 은행 등록 · 수정은 `canWrite`, 거래 수정은 `canWrite`, 삭제는 `canDelete` |

---

### 2. 구현

- 7개 파일 모두 `@/hooks/useRole`에서 `useRole` · `canWrite` · `canDelete`를 가져와 컴포넌트 최상단에 `const role = useRole();` 추가
- 등록 · 추가 버튼은 `{canWrite(role) && (...)}`로 감싸 employee 역할에서 숨김
- 상태 변경이 가능한 셀렉트(입금 · 매출채권 등)는 `canWrite(role) ? <select> : <span>`으로 분기해 employee에게는 읽기 전용 표시만 노출
- 수정 버튼은 `canWrite`, 삭제 버튼은 그보다 좁은 `canDelete`(admin 전용)로 이중 게이팅
- 조작 열 자체가 삭제 · 수정 버튼만으로 구성된 경우(`budget`, `hr/departments`, `card`) 헤더의 빈 컬럼(`""`)과 본문의 `<td>`를 함께 조건부 렌더해 employee 화면에서 빈 열이 남지 않도록 처리
- `card/page.tsx`의 거래 내역 테이블은 조작 열 유무에 따라 로딩 · 빈 상태 행의 `colSpan`을 4 · 5로 분기
