# v11.7 구현 보고서 — 저장 액션 확인 모달과 오류 처리 일원화

**작성일**: 2026-07-21  
**범위**: HR · 회계 · 생산 · 유통 전 모듈의 등록 · 수정 액션에 확인 모달을 적용하고, 저장 실패 시 조용히 무시되던 오류를 사용자에게 노출

---

### 1. 변경 사항

| 영역 | 대상 파일 | 이전 | 이후 |
|---|---|---|---|
| HR | `leave` · `payroll` · `contracts` · `employees` · `departments`(부서 · 직급) | 버튼 클릭 즉시 저장, 실패 시 입력 폼 하단에 텍스트로만 표시 | 저장 전 확인 모달 노출, 실패 시 공통 오류 모달로 표시 |
| 회계 | `vendors` · `estimates` · `tax-invoice` · `ar-ap` · `budget` | `vendors` 등은 확인 모달 없이 즉시 저장, `budget`은 400 오류만 선별 표시하고 나머지는 무시 | 5개 파일 모두 확인 모달 적용, `budget`은 모든 실패를 오류 모달로 표시 |
| 생산 | `orders`(지시 · 실적), `items`, `bom` | 확인 모달 없음, `items`와 `bom`은 저장 · 삭제 실패 시 조용히 무시 | 확인 모달 적용, 실패 시 오류 모달 표시 |
| 유통 | `orders` · `deliveries` · `vehicles` | 확인 모달 없이 즉시 저장 | 확인 모달 적용 |
| 대시보드 | `dashboard/page.tsx`의 할 일(Todo) 추가 · 수정 · 삭제 · 완료 토글 | 4개 핸들러 모두 실패 시 조용히 무시 | 실패 시 오류 모달로 표시 |
| HR | `payroll`의 급여 자동계산(`autoCalculate`) | 계산 실패 시 조용히 무시 | 실패 시 오류 모달로 표시 |

---

### 2. 구현

- 공통 패턴: `handleSave`를 트리거 함수와 실행 함수로 분리 — 트리거는 입력값 검증 후 `setModal({ title, message, variant: "info", showCancel: true, confirmLabel, onConfirm: async () => {...} })`을 호출하고, 실제 API 호출은 `onConfirm` 내부의 `try / catch / finally`로 이동
- 저장 성공 시 모달을 닫고 목록을 새로고침, 실패 시 `catch` 블록에서 동일한 `modal` 상태를 오류 메시지로 교체(`variant: "error"`) — 확인 모달과 오류 모달이 같은 상태 변수를 공유하는 기존 `card/page.tsx`의 `handleBankSave` 패턴을 그대로 적용
- 빈 `catch {}` 또는 `catch { /* ignore */ }`로 실패를 삼키던 지점(생산 품목 · BOM 등록/삭제, 급여 자동계산, 대시보드 할 일 4종)은 전부 `catch (e: any)`로 바꿔 서버 오류 메시지 또는 기본 문구를 노출
- 신규로 확인 모달을 얻은 화면 수: HR 5 · 회계 5 · 생산 3 · 유통 3 = 16개 저장 핸들러
