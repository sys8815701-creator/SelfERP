# v11.5 구현 보고서 — 개인정보 접근 범위 세분화

**작성일**: 2026-07-21  
**범위**: employee 역할의 인사 데이터 접근을 실무 관행에 맞춰 본인 기록으로 제한

---

### 1. 변경 사항

| 페이지 | 이전 | 이후 |
|---|---|---|
| `hr/leave` | 생성 · 수정 전부 admin · accountant 전용, employee는 신청조차 불가 | employee 역할도 본인 명의로 신청 가능(요청한 employee_id는 무시하고 본인으로 고정), 조회는 본인 기록만. 승인 · 타인 수정은 admin · accountant 유지 |
| `hr/contracts` | 전체 계약서 노출 | employee 역할은 본인 계약서만 반환 |
| `hr/employees` | 목록 컬럼만 급여 숨김, 상세 조회는 급여 · 계좌 · 주소 · 생년월일 · 비상연락처 전부 노출 | admin · accountant이거나 본인 레코드가 아니면 위 필드를 응답에서 제거(0/null) |
| `hr/payroll` | 전체가 admin 전용(accountant도 차단), employee는 접근 자체 불가 | 쓰기는 admin · accountant, employee는 본인 명세서만 읽기 전용으로 조회 가능, 퇴직금 탭은 계속 admin · accountant 전용 |

---

### 2. 구현

- `core/deps.py`는 변경 없음 — `hr.py`에 `get_my_employee(business, current_user, db)` 헬퍼를 추가해 "이 사업장에서 로그인 사용자에 대응하는 Employee 레코드"를 공통으로 조회
- `_redact_if_not_self()` 헬퍼로 `EmployeeResponse`의 민감 필드를 조건부로 비움
- `create_leave`는 `require_manager` 대신 `get_current_role` 직접 사용으로 완화, employee 역할이면 요청 본문의 `employee_id`를 무시하고 본인 것으로 치환
- `payroll.py`의 `list_payrolls`만 `require_admin` 대신 `get_current_business` + 역할별 필터로 전환(생성 · 수정 · 삭제 · 계산 엔드포인트는 `require_admin` 그대로 유지)
- 프론트: `hr/leave/page.tsx`의 "+ 휴가 신청" 버튼을 전 역할에 노출, 신청 모달의 직원 선택란은 employee 역할에서 "본인 명의로 신청됩니다" 안내로 대체. `hr/payroll/page.tsx`의 전체 페이지 차단을 제거하고 급여명세서 탭은 조회 전용으로, 퇴직금 탭 · 쓰기 버튼은 `canWrite(role)`로 조건부 노출
