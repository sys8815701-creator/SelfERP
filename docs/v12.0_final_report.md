# v12.0 SelfERP 검증 리포트 — v11.1 ~ v11.7 통합

**작성일**: 2026-07-21  
**검증 범위**: v11.1 ~ v11.7 전체  
**검증 방식**: 정적 코드 분석 + 회귀 curl 테스트 + 모듈별 기능 대조

---

### 1. 배경

v10.0 검증 이후 발견된 구조적 결함 — role이 `User` 전역에 저장되어 사업장 간 권한 격리가 불가능한 상태 — 를 해소하기 위해 인가 구조를 재설계하고, 그 과정에서 함께 식별된 화면별 권한 노출 누락과 저장 액션의 오류 처리 공백을 정리한 것이 v11.x 전체의 목표다.

### 3. 인가 구조 변경 요약

| 항목 | v11.0 이전 | v11.1 이후 |
|---|---|---|
| 역할 저장 위치 | `User.role` — 계정 전역 | `Employee.role` — 사업장별 독립 |
| 역할 판정 시점 | 로그인 시 1회 | 요청마다 `X-Business-Id` 기준 재계산(`get_current_role`) |
| 개인 계정 승인 · 사업자 등록 승인 | role이 admin이면 전 시스템 대상 가능 | `User.is_platform_admin` 전용 |
| 사업장 가입 요청 승인 | 소유자만 가능 | 소유자 또는 해당 사업장 admin 역할 직원 |
| 역할 변경 API | 소유한 모든 사업장 구성원 대상 전역 수정 | `X-Business-Id` 스코프 내 `Employee.role`만 수정 |
| 플랫폼 관리자 | 개념 없음 | `is_platform_admin` 별도 플래그, 사업장 미보유 상태로도 콘솔 접근 |

v11.3에서 이 구조를 두 번째 사업장으로 실제 검증했고, v11.4에서 세 가지 가입 승인 플로우 각각의 테스트 계정을 확정했으며, v11.5에서 employee 역할의 조회 범위를 본인 기록으로 좁혔다.

### 4. 프론트엔드 권한 표시 · UX 정합성

v11.6은 `canWrite` · `canDelete`가 실제로는 적용되어야 하나 버튼이 전 역할에 노출되던 7개 화면(`accounting/ar-ap` · `tax-invoice` · `estimates` · `budget`, `hr/departments`, `production/inventory`, `card`)을 기존 `vendors/page.tsx` 패턴에 맞춰 통일했다.

v11.7은 그 위에 두 가지 문제를 함께 해결했다.

- 등록 · 수정 버튼이 확인 절차 없이 즉시 저장을 트리거하던 16개 저장 핸들러(HR 5 · 회계 5 · 생산 3 · 유통 3)에 `setModal({..., onConfirm: async () => {...}})` 패턴을 적용
- 빈 `catch {}`로 실패를 삼키던 지점(생산 품목 · BOM, 급여 자동계산, 대시보드 할 일 4종)을 전부 오류 메시지 노출로 전환

### 7. 결론

v11.1 ~ v11.7을 통해 SelfERP의 인가 모델은 계정 전역 role에서 사업장별 독립 role로 전환되었고, 플랫폼 관리자라는 별도 축이 신설되었다. 이 구조는 두 번째 사업장(정성반찬)의 실데이터로 검증되었으며, 기존 행복한 베이커리 3계정의 동작은 전 단계에 걸쳐 회귀 없이 유지되었다. 아울러 이번 재설계 과정에서 발견된 화면별 권한 노출 누락(v11.6)과 저장 액션의 확인 · 오류 처리 공백(v11.7)도 함께 정리해, 사업장 격리와 화면 단위 권한 정합성을 같은 버전대에서 마무리했다.
